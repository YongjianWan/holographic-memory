# Holo 宪法 — 判断准则层（永不过期）

> **保质期:几乎永不变。** 这份装的不是"做什么",是"凭什么"。
> 新接手的 Claude 先读这份,再读 ②方案规格、③当前状态。
> 与 ②③ 冲突时:涉及**准则**(红线/为什么/规矩)以本份为准;涉及**状态/进度**以 ③ 为准。
> 来源:原《对话层交接稿》§0-§3+§5、《改造计划》§7、《改造计划》§2(Hindsight 筛选刀)。

## 0. 母红线（一切取舍的裁决依据）

> **所有 LLM 走一次性 API 调用,系统里不留任何常驻进程。关机即文件、开机即用、无启动序列。**

`facts` 是 SQLite 文件,一组无状态检索函数挂在上面。没有 Postgres、没有常驻 embedding 服务、没有 worker、没有 cron 调度器。机器关了就是一个 .db 文件,开机直接用,无启动序列。

**这条是判据,不是口号。** 方案里八九个"砍"全从它推出:

| 被砍的                              | 凭什么砍(全部回到红线)                                                                     |
| ----------------------------------- | ------------------------------------------------------------------------------------------ |
| cross-encoder rerank                | 常驻模型 → 违红线。RRF 是官方认可的降级,停在 RRF 不是偷工                                 |
| embedding 语义召回                  | 要常驻 embedding 服务 → 违红线。用 HRR/FTS5/Jaccard 替,代价是**无语义召回(睁眼认)** |
| Hindsight 整套架构                  | Postgres/HNSW/consolidation_worker/cron 全假设机器 24h 活着 → 违红线                      |
| reflect loop / consolidation worker | 常驻后台循环 → 违红线。改"惰性定时器+关机补漏"                                            |
| 自适应预算 / token 截断             | 过早优化,没遇到瓶颈(这条不是红线,是实用主义准则)                                           |

> **"为什么不用现成的 anacron"**(会被反复问,先答死):anacron 确实专为非 24h 开机的机器设计、能记时间戳开机补跑,行为正合需求。但它是**独立于 agent 的常驻调度件**,要单独配置、会失灵、平台相关——跟"无独立常驻件"红线擦边。所以定时器做进 agent 进程内(进程在跑就周期查,没跑就靠下次启动兜底),没有任何独立常驻件,最干净、跨平台。OS `cron` 更不行(到点跑、关机丢拍,正是要避开的)。

**裁决动作**:任何"加个后台服务/常驻模型/调度器"的想法,先问一句"**这个关机还活着吗?开机要不要启动序列?**"——活着或要启动,就是违红线,拦。哪怕技术上更优雅。

**红线的唯一例外不算违反**:HRR 是纯数学(SHA-256 确定性相位 + 循环卷积),不调任何模型,本身符合红线。别误伤它。

## 1. 两条正交主线 + 依赖结构

| 主线                          | 解决什么                                    | 依赖                                    |
| ----------------------------- | ------------------------------------------- | --------------------------------------- |
| **A. 检索质量(RQ/RRF)** | 每次 search 返回准不准。读路径,日常高频     | **独立,可随时插队先做**           |
| **B. 库卫生(P0/P1/P2)** | 库会不会随数据变大退化成垃圾场。写路径+维护 | **内部 P0→P1→P2 硬顺序,不可跳** |

- **A 改读时排序算法**,跟库干不干净无关 → 可最先做,不必等 GC。
- **B 内部有物理依赖**:先有干净的 fact 才有干净的图。在垃圾库上建图,边跟着垃圾平方膨胀,比不建更糟。entity 是 HRR 向量的一半成分,entity 碎裂则 HRR 检索和 P2 的边全脏——P1 不做好,P2 全白做。
  **裁决动作**:问"下一步做 X 还是 Y",先归类属 A 还是 B、前置依赖满没满足。

## 2. Hindsight 筛选刀（反复用这把刀）

> **Hindsight 给参数和判据(它拿百万级 fact 调出来的,可信),不给架构(单用户本地库养不起)。**

- **✓ 可抄**:常数、阈值、判据。`tanh(shared×0.5)`、近重复 cosine 阈值、recency 衰减、RRF `Σ1/(60+rank)`。
- **✗ 不抄**:PostgreSQL/pgvector/HNSW、cross-encoder、常驻 consolidation worker、四路检索里的 embedding 那路。**全是常驻重活,违 §0。**
  **裁决动作**:新 Hindsight(或任何外部参考系统)文档进来,逐项分"参数/判据(可抄)还是架构(不抄)"。看着好但要常驻的一律砍。

## 3. 四个被压成结论的"为什么"（丢了推理会重蹈覆辙）

**① tanh 而非 sqrt 归一化 entity overlap**
`[Aiden]` 几乎挂在每条 fact 上(整库都关于他)。原始共享数无上界,高频实体会把全库连成一团。`tanh(shared×0.5)` 自然饱和(1→0.46, 2→0.76, 3→0.91),把超级节点的拉扯压住。**别让谁改回线性或 sqrt。**

**② P0 探重必须在 INSERT 之前,不是在 except 里**
近重复(措辞不同的同一事实)**不触发 IntegrityError**,进不了 except。v1 把探重写在 except 里,结果近重复全漏进库。探重要在 INSERT 前主动查(FTS5 粗筛+Jaccard),except 只留精确撞车兜底。**看到有人想把探重挪进 except 就拦。**

**③ 禁止删库重建(铁律,无例外)**
`workspace-bridge` 敢在 schema 不匹配时删库重建,因为数据能重新 parse。**Holo 不能学——facts 是唯一来源、不可再生,删库 = 记忆永久丢失。** 任何 schema 变更走真 migration(ALTER/建新表/数据迁移),绝不 DROP+CREATE,动手前先 backup .db。软删除只走 `merged_into` 指向 dummy 999999,绝不物理 DELETE。

**④ HRR 是弱信号,修对齐是"从负资产修成弱正资产",别期待它变强**
HRR 用 bag-of-words 编码,天花板本来就低。search 的 HRR bug 修完后,query 对齐到 content 成分,但 fact 向量是 content + 多 entity `bundle` 的叠加,similarity 会被 entity 成分稀释(bundle 容量 `O(sqrt(dim))`)。修复是"从废分变弱正信号",不是强信号。**别有人看 HRR 权重低就想加权或上 embedding——加 embedding 就违红线了。**
可量指标:`snr_estimate` 在 dim=1024 时单条 fact 的 content 词数+entity 数超约 256 item,SNR 跌破 2.0 打 warning。这是"提炼粒度太粗"的客观信号。

**⑤ 反馈惩罚重于奖励,是故意的**
`fact_feedback` 非对称:helpful +0.05、unhelpful -0.10,扣分是加分的两倍。这不是手滑,是设计——记忆系统里"召回了不该召回的"比"漏召回"更伤体验(乱插嘴 > 想不起),所以惩罚要重。**别有人看着不对称就"顺手"改成对称的 ±0.05。** 同源一条审美:trust 衰减地板 0.1 也是宁可慢、不归零(留翻盘余地)。

## 4. 不可逆性不对称（贯穿所有"做不做"决策）

> **宁可漏一个该做的合并,不可做一个错的合并。** 错合并是不可逆的语义损伤;漏合并只是不完整的清理,以后能补。

同一原则推广到:schema 变更、物理删除、scope 赋值。任何不可逆动作,在证明它正确之前不做——这就是 Gate A/B 存在的理由(见 ③当前状态)。

## 5. 协作规矩（这套是 centrifugal,不是应声虫）

Aiden 要的不是越调越贴合的应声虫,是有反向力、该拍板让他拍、该拦就拦的协作者:

- **不确定就说不确定**——"我不知道"是完整句子。核出来的标核出来,不确定的标不确定,别装确定。
- **方向有问题先说,不默默执行**——发现文档打架/状态过期,先把矛盾摆出来,不顺着错的往下做。
- **过早优化要砍**——没遇到瓶颈的优化(自适应预算、为不存在的规模做的性能优化)一律砍,记 TECH_DEBT 不进路线图。
- **该他拍板的别替他拍**——把判断要素摆清让他定,别因为能接就接。
- **强观点弱持有**——给倾向但标"这是倾向不是拍板",让数据有推翻判断的权利。
- **主动拦膨胀**——他常想"顺手多做一点",而方案审美恰是"拒绝膨胀"(P1-4 三约束焊死防它变"每天写感悟"的吃灰装置就是例子)。**该拦膨胀时拦,哪怕他没要求。**

> 剥皮刀审美:抽结构,不抽人。消除边界 > 增加判断,删除比新增更让人舒服。

## 6. 进库裁决：Holo 存什么、不存什么（门口的闸）

> 这一节和 §0 母红线同性质——是"凭什么拦"的裁决依据,不是"做什么"。
> 每次有人(包括未来的 Aiden 自己)想往 Holo 塞新类型的东西,先过这把尺子。
> 来源:一次澄清式对话,把一直隐含、从没明写的两个过窄/过宽假设揪出来焊死。

### 6.1 唯一判据:有没有"触发时间"

> **判断一个东西进不进 Holo,看它是不是"无触发时间的客观断言"。**
>
> - **无触发时间、只是关于世界的一个断言** → 是 fact → 进 Holo。
> - **有触发时间(到某时刻要激活/触发/作废)** → 是待办/日程 → 归 calendar/task,**不进 Holo**。

判据为什么是这个,而不是"关于谁"或"重不重要":Holo 的每一个机制(trust 衰减、recency、P0 探重、被动查询召回)都是为"朝后看的、已发生的、可去重的断言"造的。带触发时间的东西是"朝前看的、未发生的、到点即焚的意图"——时间箭头相反,Holo 没有任何"到点触发"机制,塞进去到那天屁都不会发生,只会被当普通 fact 沉在库里慢慢衰减。

同一个主题可以裂成两边,分界就是触发时间:

- "提醒我 6/28 走 OA" —— 有触发时间 → 日历。
- "Aiden 的 OA 流程要走部门审批" —— 无触发时间,常驻事实 → Holo。

⚠️ **时间锚定**:任何相对时间词("后天""下周三""本周内")在**说出口那一刻**就必须解析成绝对日期再存,否则它会腐烂——今天的"后天"明天读就错一天。这是 calendar 的基本功,顺带印证这类东西本就归 calendar:连"存哪天"都是日程系统的活,不是记忆的活。
(衔接 ③"有效期语义待设计":带明确失效时点的 fact 是这条的轻量版——它在 Holo 里但需要"过期"语义;纯待办是它的极端形态——压根不该进 Holo。)

### 6.2 松绑:fact 不限于"关于人"(删掉一个从没明写的过窄假设)

Holo 早期隐含假设"主要存关于 Aiden 的事"。**这条假设作废。** Holo 存 **fact**,fact 可以关于人、关于物("大理石轴矮轴机械键盘不好用")、关于项目("P2 因 fan-out 0.811 被否决")、关于任何东西。§3① 里 `[Aiden]` 是高频实体只是**碰巧**(整库关于他),不是"只能存他"的规定。

判据永远是"是不是 fact"(见 6.1),**永远不是"关于谁"**。

### 6.3 Holo 是一个器官,不是整个身体(反膨胀总闸)

> **一个完整的 AI 助手需要多个器官;Holo 只是其中的"长期记忆"那一个。守窄才配得上未来,不是装得越多越配得上未来。**

反复出现的膨胀模式:想到助手该有的新能力(待办、提醒、过程日志、感知、主动循环),就习惯性想把它塞进唯一已造好的器官——Holo。**手里只有一把锤子,看什么都像钉子。** 正确动作不是扩张 Holo,是问"这是哪个器官的活",分流过去:

| 想塞的东西                              | 它的真实性质           | 该去的器官            | 是 Holo 吗                         |
| --------------------------------------- | ---------------------- | --------------------- | ---------------------------------- |
| 键盘/项目/决策/踩过的坑(无触发时间断言) | 记忆(朝后看)           | **Holo (fact)** | ✅                                 |
| 操作手册、给 AI 的指令(**逐字原文**,要被当指令执行) | 逐字稳定的指令 | SOUL / preferences    | ❌(转写会漂移,见 summary spec §9) |
| changelog、过程流水(**逐字原文**,要按时序查"谁、何时改了什么") | 不可去重的时序日志 | git / 文件            | ❌(撞 P0 探重)                     |
| "后天提醒我走 OA"、日程                 | 待办(朝前看、到点即焚) | calendar / task       | ❌(见 6.1)                         |
| 实时"连敲四小时了"                      | 环境感知流             | 感知层(不入任何库)    | ❌                                 |

### 6.4 例外:项目自身元文档经原子提炼后可入库(2026-06-27 补,Aiden 拍板)

> 6.3 表里"操作手册/changelog 不进 Holo"原意是**禁止把逐字原文当指令/日志存进 facts 表**——那种存法既会漂移又会撞 P0 探重。**它没有禁止"对这些文档做和任何其他文档一样的原子事实提炼"。**

宪法.md、AGENTS.md、CHANGELOG.md、ROADMAP.md、SESSION.md、TECH_DEBT.md 等项目自身元文档,经 `retain_document` 走标准原子提炼管线后,抽出的每条 fact 本身是"无触发时间的客观断言"(某次决策为什么、某个方案的判据是什么),符合 6.1 判据,**可以入库**,不被本节当反例拦。

- **入库的是提炼出的原子断言,不是文档原文**——原文仍然只活在 git/文件里,不被当成"逐字指令"或"逐字日志"复制进 facts。这条区分让本条不与上面 6.3 表冲突。
- **裁决动作**:看到"项目元文档要不要进 Holo",先问"是存原文(逐字)还是过提炼管线(原子化)"——原文不进,原子化结果可以进。

**为主动 AI 的未来做准备 = 让 Holo 的 fact 更纯、更可信,不是让 Holo 能力更全。** 越主动的未来,对事实准确度要求越高——主动 AI 拿 fact 当"开口的依据",基于错 fact 主动来烦人是灾难。一个混着待办、日志、噪声的 Holo,喂给主动 AI = 它自信地、反复地拿垃圾烦你。**装得越杂,主动的未来越是灾难;守得越窄,越多别的器官敢信任地调用它。** 窄不是局限,是它能被信任的前提。

> 与 §0 同源:§0 拦的是"加常驻件",§6 拦的是"加器官"。两条都是边界焊死,都服务于"Holo 是什么"的纯粹性。

## A 线 — RQ:RRF 默认融合三路（独立可先做,已 done）

默认 search 融合三路(FTS5 / Jaccard / HRR):它们量纲/基准不可比,用 **RRF** 融合,只用名次不用原始分:

```
score(fact) = Σ_i  1 / (k + rank_i(fact))      # k = 60
```

- 每路各自排序取名次,多路靠前者得分高(共识)
- k=60 平滑,防头部垄断
- 纯算法、零依赖、零 LLM
- RRF 出排序后叠一层 trust/recency 的**乘法** boost(中心 1.0、±10% 限幅)。乘法保证次要信号压不过主排序——加法会让勉强相关的靠 recency 盖过高相关的。
- **降级合理性**:不上 cross-encoder(常驻,违红线),停在 RRF 是 Hindsight 官方认可的降级。
- **HRR 必须保留在默认 search**:2026-06-29 固定查询 A/B 显示,HRR 会显著改变排序(median top5 overlap 0.8,20 条里 12 条 top1 改变)。在无 embedding / 无语义召回约束下,HRR 是唯一可行的本地弱语义/结构信号,不能因为它弱就踢掉。HRR 不是词义等价表;词语等价走 SQLite 本地 `semantic_equivalence_*` 表扩展 query,不引入常驻服务。

## 输入侧 — 文章入口 + 存原文（已 done）

```
retain_document(raw_text):
    facts = LLM提炼(raw_text)        # 一篇→多条原子 fact,一次性 API
    for f in facts: add_fact(f)       # 各自走 P0 探重
```

- 提炼走**原子事实**(细),粒度粗则 P0 探重和 P2 建边都失准。
- 存原文:`documents` 表(doc_id/raw_text/source/created_at)+ `facts.source_doc_id`。recall 默认返回 fact,要"回看完整"按 source_doc_id 拉原文。
- ⚠️ 原文不可再生,存前 backup,走真 migration。

## B 线 P0 — 写入探重（已 done）

在 `add_fact` 的 **INSERT 之前**插 `_find_near_duplicate` + `_merge_into`,原 except 块保留作精确兜底。

三条硬约束:

- **A. FTS5 粗筛 + Jaccard 精判,不用 HRR**(entities 缺席 + 同类信号冗余)。语义近重复故意留给 P1。
- **B. 探重的 FTS5 查询必须删 `retrieval_count += 1` 且去 trust 过滤**(否则污染 GC 信号 / 漏低分重复)。
- **C. merge 改写 content 防 UNIQUE 冲突。**
  merge 选哪条留下:本地打分不掏 LLM,`score=(实体数+数字/版本/日期命中)/log(len)`。merge 只解决【重复】不解决【烂】,质量归 P1 trust 衰减。

## B 线 P1 — 批 GC（触发机制见 ③,已被并发补丁修订）

**P1-1 entity 归一化(已 done)**:编辑距离 + HRR 相似度聚类近义节点,合并保留主节点、其余进 aliases、重指 fact_entities 外键。+ 数字/版本签名闸防 K2/K2.7 误并。**这是 HRR 和 P2 的地基。**

**P1-2 语义合并 + 时间线收敛(规格定,阈值待标)**:P0 只挡词面,两类漏到这:措辞不同的同一事实、非字面矛盾的时间线(Mia 演化史 → 收敛成"当前态+演化史",保留演化史不只留终态)。两段式:HRR 相似度粗筛 + 一次 LLM 读全文细判。⚠️ Hindsight 的 0.97 cosine **不能直接搬**,HRR 分布不同,真实数据上重标。

**P1-3 trust 衰减(规格定)**:`recency=clamp(1-days/365, 0.1, 1.0)`,地板 0.1(再老不归零,留翻盘余地)。依赖 P0-B 的干净 retrieval_count 信号。

- 衰减后**低于 trust 删除阈值的**:list/search 不返回,或 GC 时 remove。删除阈值是四个待标阈值之一,真实数据上标。
- ⚠️ 地板 0.1 与删除阈值不矛盾:地板防的是"recency 这个乘子"归零(乘法链里一项归零全归零),删除阈值作用在**最终 trust** 上。两个是不同层的闸,别合并。
  **P1-4 跨话题串联** — 见下方独立规格,gated on Gate A。

## P1-4 规格（gated on Gate A,Gate A 不过则整块作废）

> 规格定型,实现路径未变(惰性 GC + 一次性 API + 出口闸)。**没有新组件。**
> 来源:拆 Honcho 文档,只取维度切分与判据,不抄实现(它全踩红线)。

- **只做 induction**(跨领域结构相似性识别)。不做 deduction(造新 fact 断言,与"删不加"审美冲突),不做 abduction(揣测动机/心理,撞人格红线)。
- **`source_fact_ids` 非空闸**:每条 observation 必须带 fact id 列表,为空则拒绝入库。把"禁止漂亮废话"从 LLM 自觉升级为 schema 硬拒绝。起点 ≥2(单条不构成"跨"),真实数据上标。
- **禁止动机揣测**:只陈述"哪几条决策/偏好结构同构",禁止"因此他是怎样的人/他为什么这样"。抽结构不抽人。
- **粒度双向参照**:输入端攒 ~1000 token 块再喂 LLM(Honcho 数,起点参考),输出端单条 fact 压在 SNR warning 线下。
- **质量天花板诚实认**:通用模型做归纳会比自训模型更易产"漂亮但逻辑松"的 observation。防线是出口闸(source_fact_ids 非空),不是源头模型。**产物糙到不能用 = 这条路在无依赖约束下不成立,该砍 P1-4,不是该加依赖。**

## 同义/黑话 lexicon — P1-2/P1-4 的副产派生表（规格定,产出 gated 继承上游）

> 来源:2026-06-29 决议补丁。解决 FTS5/Jaccard 词面匹配的同义盲区(母红线"无语义召回"的具体疼点),但**不引入任何向量/embedding**。
> 一句话:**同义关系的箭头,黑话部分只有 LLM 读你自己语料才填得出;词频统计能圈出"哪些词是高频黑话",填不出"它的同义词是什么"。** 故让 P1-2/P1-4 那趟已有的 LLM 调用顺手副产同义映射,不新增组件。

**四条规格:**

1. **存哪:独立派生表,非 fact。** 用库里现成的 `semantic_equivalence_groups` / `semantic_equivalence_terms`(v11 本就为此留,当前 0 条)。
   - 与 summary 同待遇:**不进 facts 表、不被 P0 探重、不被 trust 衰减、不参与 RRF 排序竞争、可丢可重建。**
   - 铁律③不咬它:lexicon 是 fact 的派生投影,源料(产它的那些 fact)都在 → 投错重投。删了能重建。
2. **谁产:P1-2 收敛 / P1-4 归纳那趟 LLM 调用的副产物。** 不新增 LLM 调用、不新增组件、不碰写事务、不撞并发补丁①。
   - LLM 细判两条 fact 关系时,顺手回一句"这俩用了不同词指同一概念:X/Y",落 lexicon 表。
   - 为什么非挂在 P1-2/P1-4 上:**黑话同义箭头的唯一可能来源是"LLM 读你自己的语料"**——word2vec 对黑话瞎(训练语料没你的词)、词林对黑话瞎(通用词典没你的词),只有读真实语料的那次 LLM 调用填得出"探重=去重"这种自造词的桥。
3. **怎么用:查询时同义扩展,低权重兜底。** query 里的词命中 lexicon → 扩展同义词进 FTS5 召回。
   - ⚠️ **扩出来的词给低权重,不与原词在 RRF 平权竞争**——套 §A 线"乘法 boost ±10% 限幅"的审美。否则同义噪声盖过精确命中(同 §3① 怕的"高频/杂义把全库连一团"病,只是发生在词面层)。
4. **严格 gated,继承上游:** P1-4 gated on Gate A;P1-2 生产策略 gated on 四个阈值标定。**lexicon 是它俩的副产物,自动继承这俩闸——它俩没解禁,lexicon 没原料来源。**
   - ⚠️ **不许为了早点拿 lexicon 把 P1-2/P1-4 提前放开。** 那是拿未验证机制喂新需求,典型 runs ahead。

**唯一现在就能动、不等任何闸的种子来源:** 召回审计手标。跑审计把 miss 捞出来、手标几十条黑话同义对 → 填 lexicon 当初始种子(不依赖 P1-2,是手标)。这批手标还能当 P1-2 解禁后的**验证基准**(LLM 碰撞出的同义对跟手标对不对得上),并顺带回答"黑话到底有多少"。

> ⚠️ **命名钉死:这不是"边"/"共边"。** 它是"LLM 在合并/归纳时副产的同义映射",进 lexicon 表,**不进任何 edges 表**。叫"共边"会让未来误以为在复活 P2,撞坟。

## Summary 层 — fact 的派生投影,对抗条目爆炸(gated on Gate B + 前置 P1-2)

> 来源:与 Aiden 对话(scope 灵感源自 Claude project 架构;实体粒度收敛对照 Hindsight 的 Mia demo)。
> **双重 gating**:整体 gated on Gate B(scope 不成立则容器不存在,整块作废,与 P1-4 gated on Gate A 同待遇);**且前置依赖 P1-2 时间线收敛**(脏 fact 上压不出干净 summary,理由见下)。解禁后顺序 **P1-2 → summary,不可颠倒**。
> 一句话:**只有原子 fact 没有 summary 的记忆是破碎的**——fact 是原料,summary 是叙述骨架(因果+时序+演化轨迹)。读一堆孤立 fact 拼不出来龙去脉。

### 0. 存在理由(统摄本块所有设计):对抗条目爆炸

> **summary 的本质不是"叙事化",是 fact 只增不删(铁律③)+ 无语义召回(母红线)双重约束下,对抗条目爆炸的结构性解药。**

fact 单调膨胀是本架构的宿命:留得越多,FTS5/Jaccard 信噪比越烂(HRR 饱和是其一种形态)。summary 让"读懂这个人/项目/实体"的成本**不随 fact 数增长**——fact 层继续无限留(铁律③什么都不丢),summary 把 N 条压成一份**恒长**投影。读一份 summary 抵扫全库。

后面所有设计都是这一条的推论:

- **增量演化(旧 summary + 新增)**:全量重建 = 每次扫爆炸的 fact 堆,自己被拖垮,起不到解药作用。
- **token 上限焊死**:summary 能随 fact 变长 = 爆炸从 fact 层漏到 summary 层。上限保证"恒长",**不是省空间**(容量从来不是约束,堆叠才是)。
- **提炼掉琐事**(如一次性硬件比价 fact):恒长 summary 配额稀缺,只给"真的定义此人/项目/实体"的内容;琐事留 fact 层(无限,不丢),不占 summary 配额。
  ⚠️ **summary 背不动的锅(诚实认):** summary 解的是**理解成本爆炸**(读懂的成本),不解**检索信噪比爆炸**(找到具体一条的成本)。"Aiden 追踪过哪款显卡"是检索问题,summary 已把它提炼掉,得回 fact 层 FTS5 找。**summary 替代"通读",不替代"检索"。** 检索信噪比归 A 线 RRF / 未来 scope 分桶,别让 summary 背。

---

### 1. 定位:summary 不是 fact,是派生层

- **不进 `facts` 表、不被 P0 探重、不被 trust 衰减、不参与 RRF 排序竞争。**
- **持久化**:每个粒度的每个单元存一份**当前态**(累积实体,不是读时现算的视图)。
- **可整体重算、可丢弃**:summary 是 fact 的投影,源料 fact 都在 → 投错重投。**不碰铁律③**(facts 不可再生→不可删;summary 可再生,删了能重建)。

---

### 2. 一棵压缩树(不是三种并列对象,是一棵树的三层深度)

summary 不是"三种 summary",是**一条逐层上卷的压缩链**,父节点是子节点的目录:

```
全局 summary        ← 印象层(他是谁:协作底色、判断模式、审美红线)
   ↑ 压缩(父只吃子的 summary,不下钻原始 fact)
project summary     ← 事情层(这个项目里发生了什么、怎么演化的)
   ↑ 压缩
实体/时间线 summary ← 细节层(单实体怎么演化:Mia 当前态 + 演化史)
   ↑ 压缩
原始 fact           ← 原料(一条条事实,无限留,铁律③)
```

- **通用规则:任何父节点只吃直接子节点的 summary,逐层上卷,不下钻到孙子**(全局吃 project summary,不吃 project 的原始 fact)。原 §8 的"两级"是这棵树的特例,不是另一套机制。
- 每层都是"一组下层单元 → 一份恒长派生投影,不进 fact 层,会过期就重算",仅**深度**不同。
- **实体/细节层 = P1-2 时间线收敛的产物**。Mia 四条(加入→晋升→跳槽→回归)收敛成"当前态(头)+ 演化史(四条原始 fact 子项原样保留)",那个**"当前态"的头就是细节层 summary,是派生投影,不是合成进库的新 fact**。详见 §5。

#### 2.1 渐进式披露 = 读取路径(印象→事情→细节)

层级不只是存储形状,是**读法**,同构于人的回忆方式:先想起印象(他是怎样的人),再想起事情(一起做过什么),最后才是细节(那次具体怎么回事)。不一上来 dump 所有深度。

- search 命中 → 先给**印象层**(全局画像)
- 要更具体 → 下钻**事情层**(相关 project summary)
- 要细节 → 再下钻**细节层 / 原始 fact**

> 这反向解释了 §4 的 token 上限为何可行:**每层只承担它那个深度的信息**(印象层不装细节,细节在下一层等着),所以每层能压得很短。**层级和 token 上限是一回事的两面**——没有层级,单层就得装下所有深度,必然膨胀。

#### 2.2 下钻指针:查询时实时算,不存进 summary（⚠️ 易踩坑)

下钻需要父能指向子。但**父 summary 绝不存子节点的硬指针**——指针会失效(子 project 被合并、某 fact 簇被 P1-2 收敛,存死的指针指向坟墓,这是 `source_doc_id` 单值 provenance 问题的翻版)。

- **下钻路径按 scope 归属关系查询时实时算**:全局下钻 = 列该全局下所有 **active** project scope;project 下钻 = 列该 scope 下 **active** fact。
- summary 里**只存每层那份恒长文本投影,不存目录指针**。merge/收敛随便发生,下钻路径永远是 active 状态实时算出,不指向坟墓。
- 同源原则(与 §5 当前态、与会失效关联一致):**会失效的关联别固化,查询时重算。**

---

### 3. 生成:累积增量,挂 GC 那趟

```
新 summary = LLM(旧 summary + 自上次以来该单元新增的 fact)
```

- **吃"上一份已压缩 summary + 增量 fact",输入量恒定,不随该单元 fact 总量涨。** 别误写成"读全部 fact 重建"(会随库变大喂不下,且自己被爆炸拖垮)。
- **时机:挂在 GC 那趟,惰性、锁外、抢锁跑**(见并发方案②:session 开始/retain 后/定时器醒来查时间戳,抢到锁顺手干)。**不新增机制、不碰写事务、不撞并发补丁①**(LLM 调用绝不焊回 retain 写路径)。
- **不判断"每次新 fact 是否重算"**——不判断,攒着,等 GC 那趟一块干。

---

### 4. 防膨胀闸(出生即焊死,等同 P1-4 的 source_fact_ids 非空闸)

- **输出 token 上限**:summary 长度焊死上限。压不进 → 逼 LLM 真提炼、丢次要,**不许堆叠**。这是 summary 版防膨胀闸,挡它长成"每天写感悟"流水账。
  - ⚠️ 上限具体数 = **起点值,真实数据上标**,与四个待标阈值同待遇。
- **无独立管理语义**:不存历史版本、summary 间不建关联、不自动定时刷新(刷新只搭 GC 便车)。任何"summary 自己也要被管理"的想法 = 膨胀,拦。

---

### 5. P1-2 时间线收敛:只收敛,不合成 fact（⚠️ 此处需回改 ② P1-2 既有段落 + ① §3③ 旁红线重述）

Hindsight 的 Mia demo 演示的"弄明白当前态",**做法是合成一条新断言"Mia 当前=principal ML engineer"——这正是本项目 §3③/P1-4 禁的 deduction,且会过期。直接照搬撞两条红线。**

**正确做法(路 A,锁死,无路 B):**

- 收敛 = 把矛盾 fact 标成**有序演化链**(按时间戳),**保留全部历史 fact**(铁律③,一条不删)。
- **"当前态"是实体粒度 summary(派生投影),查询/GC 时由"读演化链最新"算出,绝不作为新 fact 写进 `facts` 表。** Mia 再跳槽 → 重算这个派生头,旧头被覆盖,无"删 fact"动作(头从来不是 fact)。
  ⚠️ **没有"路 B"(合成当前态入库)这条活路,也别拿"打包/上级目录"绕:**
- 把当前态当"上级目录的头",那个头若是 fact → 只是给会过期的合成断言套了层壳,铁律③照样焊死一条会错的;若不是 fact → 就是路 A 换名字。**"打包"在"头进不进 fact 层"这唯一要害上不提供新答案。**
- "演化史保留 + 当前态明确"的效果,= P1-2 收敛(留链)+ 实体粒度 summary(当前态作派生头),**不往 fact 层合成任何新断言**。
  ⚠️ **deduction 红线精确重述(改 §3③ 旁措辞,准则层改动,待 Aiden 拍):**

> **禁的不是 "deduction 这个动作",是 "deduction 的产物进入不可再生的 facts 源料层"。**
> summary(含实体当前态)做同样的合成动作,但产物去派生层、不进 facts 表、可丢弃,故不受此红线约束。红线焊在**产物去向**,不焊在动作。

---

### 6. 统一原则:派生层"先做随便修",fact 层合成"必须 gated"

> **"先做出来再修、反正 fact 存着不丢"——成立的前提是产物留在派生层。**

- 产物在**派生层**(summary / 实体当前态 / 查询时算的东西)→ **先做,随便修,鼓励先做**(还能在真数据上标阈值,空库上调=拍脑袋)。可逆性来自"源料 fact 还在"。
- 产物进 **fact 源料层**(合成断言)→ **先做 = 往不可逆的层塞还没验对的东西 = runs ahead**(GC 空库先建、P2 空库差点建,都是此病)。铁律③在这里反向咬人:fact 不可删,会错的合成断言塞进去就**永久不可逆**(撞 §4 不可逆性不对称)。
- **一句话**:铁律③不是"所以怎么做都能修",是"**所以更不能往 fact 层塞会错的东西**"。

### 7. 检索牵带:summary 随结果返回(上界焊死)

- summary 不进 RRF 竞争,但 search 命中 fact 后,按命中 fact 所属 scope **牵带**返回——不靠相关度打分(summary 对 scope 内任何 query 都相关,靠打分反而在措辞不重合时漏掉)。**summary 是"语境头部",被命中结果牵带,不是被检索出来。**
- **数量上界:最多 2 个 project scope + 全局 = 3 份。** 非性能妥协、非待标阈值,是**设计决定**——人一次装不下三个以上语境的来龙去脉。命中超 2 个 project 时按命中强度(命中 fact 数/rank)取前 2,其余只留"还命中了 scope X(summary 未附)"标记。
- **全局 summary 无条件垫底**,不参与上面取舍、不靠"是否命中全局 fact"决定。全局装 Aiden 的底色,对任何 project 语境都该在场——这是本职,不是硬塞。稳态:**头部永远至少全局 1 份,最多 3 份**。
- 反例警示:只能"专门去读"、search 时不出现的 summary = 把来龙去脉锁进抽屉。破碎恰恰发生在"拿到一堆 fact 却不知它们为何在一起"那一刻,summary 必须在那一刻在场。

---

### 8. 全局 summary 的输入构成(压缩树顶层的料从哪来)

压缩树通用规则(§2:父只吃子的 summary)落到顶层的具体构成。参照 Claude 自身记忆架构:全局记忆与 project 记忆是**两个独立的池子**,全局不汇总 project 的原始 fact。

- **全局 summary 输入 = 全局层自己的散 fact(不属于任何 project、关于 Aiden 这个人的)+ 所有 project 的当前 summary。**
  - 全局**不吃各 project 的原始 fact**(爆 token + 重复劳动),吃 project 已压好的 summary(= §2 父只吃子 summary)。
  - 两级提炼,token 天然收敛。
- ⚠️ 由此 **scope 不是严格树**:project summary 双重身份——既是 project scope 的终产物,又是全局 summary 的输入原料。非 bug,显式写明免后续误判其归属层。

---

### 9. 全局 summary vs project summary:抽人 vs 抽事件(本质区别,非粒度区别)

- **project summary = 事件的叙述**:scope 内发生了什么、怎么演化、为何现状。有时间线、绑具体内容、**会过期**(项目结束即归档)。
- **全局 summary = 人的画像**:Aiden 是怎样的协作者、判断模式、审美与红线、反复显现的特质。抽离具体内容、**底色比任何单个项目活得久**。说的不是"在某 project 里做了什么",是"是个会这样做事的人"。
- **这解释了为何全局无条件垫底而 project 不**:底色是跨 project 的常量,project 演化史只对自己 scope 适用(搜 A 垫 B 的演化史是噪音)。
  ⚠️ **抽人边界 = "有观测支撑 vs 脑补",不是"行为 vs 动机"**(改我最初焊死的措辞):
- 合法:有反复观测支撑的模式概括,**即便是动机性的**——如"靠不可见进度条续命""被甲方关系+求稳性格按住"(都有具体行为反复作证)。
- 非法:无观测支撑的脑补——"他因为童年某事所以回避"(无证据、纯推测,= abduction)。
- 即:全局 summary 抽**有观测支撑的画像**(含模式与倾向),不抽**无支撑的脑补动机**。与 §3/P1-4"抽结构不抽人/禁 abduction"不打架,边界在"可观测 vs 揣测"。
  ⚠️ **全局内部快/慢过期分层**(全局不是铁板一块):
- 慢底色(几乎不过期):协作风格、审美、Linus 操作框架、贯穿性特质。
- 快变量(随时会过期):当前薪资、转正时点、在追的赛事、手头 offer。
- 增量更新须让**快变量可被覆写/淘汰**(旧的"备考某考试"该随毕业失效退场),慢底色保持稳定。否则全局 summary 会钉死过期信息当噪音垫在 search 头部。
  ⚠️ **不设情感/关系排除区(看似该设,实则有害):**
- 直觉上情感/关系/心理脆弱性该排除出无条件垫底的全局画像(隐私 / centrifugal 的"该往外推")。**在有精准语义召回的系统里这成立**——情感信息躺 fact 层,问才召回。
- **但 Holo 母红线砍掉了语义召回("无语义召回,睁眼认")。** FTS5/Jaccard 词面匹配,语境相关但词面不沾的 query 召不回这类 fact。踢出 summary 只留 fact 层 = 交给召不动它的检索器 = **实质丢失**。
- 故:**无可靠召回前提下,全局 summary 无条件垫底是这类信息唯一能稳定在场的载体,不设排除区。** 排除区的前提(精准召回兜底)在 Holo 不成立。
- centrifugal 边界澄清:它管 **AI 主动该不该往外推**(情感话题该说"找真人,陪想不替代"),不管 **记忆存不存/垫不垫**。无条件垫着 ≠ 主动推送;知道得全 ≠ 做得越界。
  ⚠️ **summary 只装画像,不装操作指令:**
- "别夸他东西多""防三种失败模式"这类是**给 AI 的操作指令**,不是"关于 Aiden 的事实",归 **preferences/SOUL 层**(本就在那)。
- 不进 summary 的 LLM 增量转写——事实可被转写,**指令不能**(转写几轮会漂移:"别夸他东西多"可能歪成"对其项目保持中立",意思就变了)。指令需逐字稳定。

### TECH_DEBT(不进主 spec,等咬人再修)

- **summary 漂移**:每次只吸收增量、旧叙述被反复转写,早期信息可能慢慢失真(复印件的复印件)。修法:低频"从全量 fact 重建 summary"做校准。**现在不做**(实用主义:等观察到漂移再上),风险记此,别假装没有。

### 不做(scope 数量伸缩性)— 个人场景下不存在此问题

- "N 个 project summary 全喂进全局 → 爆 token" 是**企业/多租户**问题,**不是个人单用户问题**。个人场景每周活跃 scope 个位数、project 总数长期个位数到十几,全局吃这个量不会爆。
- **直接判定此问题在当前 Holo(个人/单用户/本地)下不存在**,不留 TECH_DEBT、不预设活跃度筛选 / meta-scope 分层。为不存在的规模做设计 = runs ahead,违母红线。真变多人/企业用 = 另一个产品另一条红线下的事。
