# 并发补丁 — 多 agent 共享同一个 .db

> **这份是给 Kimi 当施工图的设计文档,不是代码。** 判断错一处,Kimi 会忠实地把错的实现出来,所以每条结论都标了「凭什么」。
>
> **修订关系**:`记忆插件改造计划` §5 把 GC 触发焊死为「**hermes 进程内**惰性定时器」。那条假设建立在「只有一个常驻进程碰库」之上,**已被实际用法打穿**——Aiden 的真实用法是同时开 N 个 Claude Code 实例处理不同的事,N 不固定,且**都往同一个 `memory_store.db` 读写**。「单进程独占」从没被显式写下,但它隐含在几乎所有写路径里。本补丁把这个隐含假设拆掉。
>
> **母红线一条没动**:库还是本地 SQLite 文件、无服务端、无常驻调度件、LLM 还是一次性调用。本补丁是「关机即文件」的延伸——**库没有持有者,它就是躺在那的文件,谁需要谁开**(路 A)。

## 0. 三层决策,别混(这几轮踩过的混淆)

「让 Holo 通用」有三层,从小到大,**上层不蕴含下层**:

| 层               | 含义                                                                                                              | 状态                                                              |
| ---------------- | ----------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------- |
| **第一层** | Holo 给 Hermes 用                                                                                                 | 已在做,无需决策                                                   |
| **第二层** | Holo 抽成**独立包**(无状态函数 + SQLite 文件),Hermes / Scout / N 个 Claude Code 都 import、各自开同一个 .db | **本补丁的前置,必须先做**                                   |
| **第三层** | 这些框架**真并发写**同一个库                                                                                | Aiden 已确认会发生(开多个 Claude Code 写同一 `memory_store.db`) |

**关键**:第二层(代码可复用)≠ 第三层(真并发)。第二层几乎免费——Holo 架构本就是无状态函数挂在 SQLite 上,没有「属于 Hermes」的耦合,抽成独立包是一次性整理,不动红线、纯收益。**第三层才需要付并发的工程账**,而 Aiden 已确认第三层为真,所以这账要付。

**前置警告(给 Kimi)**:本补丁所有改动假设 Holo 已抽成独立包、`MemoryStore(path)` 能被任意框架打开同一 .db。如果这步还没做,先做这步,再动下面的并发改造。

## 1. 并发模型定性

**路 A:多进程直接打开同一 .db 文件,SQLite WAL 多读单写托底。**

- **读并发零成本**(WAL 原生支持多读)。
- **所有真问题都在写并发**——只在多个实例同时 `retain` / `add_fact` / GC 时发生。

**为什么不走路 B(一个 agent 持有库、其他走它的接口)**:路 B 能让写并发整体消失(只有一个进程碰库),但持有库的 agent 变成「必须先启动的记忆服务」,其他 agent 依赖它活着 → 「无启动序列」「无常驻进程」双破。Holo 命根子是无启动序列,路 B 违红线,不采用。**路 A 是唯一符合红线的多 agent 方案**,代价是显式处理写并发——付得起,不动架构。

## 2. 三处写并发改动(按急迫排)

### ① retain 拆事务 + P0 探重进事务 —— 最急

**问题 A(超时/死锁)**:`retain_document` 里 LLM 抽取要几秒。若这几秒握在写事务里,N 个实例一起 retain 就串行排队几秒×N,`busy_timeout` 直接超时。

**问题 B(P0 探重竞态)**:`add_fact` 的 `_find_near_duplicate` 是 check-then-act。实例 A 查「有无近重复」→无→准备 INSERT,同一瞬间 B 也查同一条→也说无→也 INSERT,两个都漏过探重,库里进一对**近**重复。`content UNIQUE` 只挡**精确**重复(第二个 INSERT 撞 IntegrityError),挡不住近重复——而近重复正是 P0 存在的全部理由。

**两个问题一个解,形状如下**:

```
retain_document(raw_text):
    facts = LLM抽取(raw_text)              # 锁外。几秒。N 个实例并行抽,互不挡
    with 短事务(BEGIN IMMEDIATE):           # 锁内。毫秒级
        for f in facts:
            dup = _find_near_duplicate(f)   # 探重移进锁内:拿到写锁后才探
            if dup: merge(dup, f)           # 探+写原子,竞态消失
            else:   INSERT(f)
```

**为什么 B 这个场景必须补、不能「交给 P1」**(这是我中途收回重判的一处,记牢):

- Aiden 的用法是**并行灌相关文档**(几个 Claude Code 各处理一批相关资料)。相关文档之间近重复率本就高,N 个实例同时 retain 时这个竞态**不是偶发、是高频命中**——并行越多、文档越相关,漏得越凶。
- 「P0 不求完美,交给 P1」这个定位是给**单写者偶发竞态**的。套到「高频并发写相关文档」上,漏过率差一个数量级,**不成立**。

**为什么探重进事务不破坏拆事务的目的**:

- 拆事务要赶走的是「**几秒**握锁」(LLM 抽取)。探重是 FTS5 查一下 + Jaccard 算一下,**毫秒级**。
- 探重进事务只让写锁多握几毫秒,跟 LLM 的几秒不是一个量级。LLM 抽取仍在锁外并行。
- ⚠️「探重进事务会拉长写锁、与拆事务冲突」是把 FTS5 查询的几毫秒错当成 LLM 的几秒——**不成立**。两件事一起做,不矛盾。

**给 Kimi 的核对项**:动手前先看 `retain_document` / `add_fact` 的真实事务边界——LLM 调用现在是否夹在 `BEGIN...COMMIT` 间?`add_fact` 现在是显式事务还是 autocommit?照真实结构改,别照本文件的伪代码硬套。

### ② GC 归属 —— 被用法打穿的设计漏洞,数据正确性

**改造计划 §5 的「hermes 进程内定时器」在这个用法下不成立**:Aiden 开的是 N 个 Claude Code,没有一个叫 hermes 的常驻进程。两种坏情况二选一:

- 没有任何实例自认为该跑 GC → **GC 永不触发**,trust 衰减/语义合并全不发生,库慢慢腐烂。
- 每个实例都触发 → **并发改同一批 fact 的 `merged_into` 和 trust** → 软删除写冲突,且错了不一定立刻看得出来。

**解法(不违红线):GC 抢锁,谁抢到谁跑,抢不到就跳。**

- 触发点从「hermes 专属」改读为「**任一活着的实例**,在 session 开始 / retain 之后 / 进程内定时器醒来时,都去查时间戳」。叠加越多越不漏(本就是 §5「惰性定时器+补漏」的精神)。
- 真正跑 GC 前用 `BEGIN IMMEDIATE` 抢写锁。抢到→跑;撞锁(别人正在跑)→**立刻退,不等**。
- **为什么「抢不到就跳」零代价**:GC 幂等 + 可延后。这一拍跳了,下次哪个实例醒来发现「距上次 GC 超阈值」再补,正是 §5 容忍关机丢拍的同一机制。多 agent 抢锁失败 ≈ 关机丢拍,补漏逻辑原样适用。
- 锁的载体:直接用 SQLite 的 `BEGIN IMMEDIATE` 写锁最干净(无额外状态)。或 `gc_log` 表加一行「跑 GC 中 + 时间戳」。

> **对改造计划 §5 的修订声明**:§5 中「hermes 进程内」「hermes startup」等措辞,在多 agent 用法下一律改读为「**任一活着的实例**」。GC 不再属于某个特定进程,它属于「谁先醒来且抢到锁」。**无独立常驻调度件这条红线不变**——定时器仍是各实例进程内的,没有新增任何独立于 agent 的常驻件。

### ③ busy_timeout + checkpoint 降级 —— 最轻

- `PRAGMA busy_timeout = 5000`(5–10 秒):N 个实例排队,锁等待要够长才不偶发超时。
- `close()` 里的 `PRAGMA wal_checkpoint(FULL)` **降成 `PASSIVE`**(或交给 `wal_autocheckpoint`):`FULL` 会等所有读者读完才 checkpoint,多 agent 下一个关库时另一个在读会卡;`PASSIVE` 能 checkpoint 多少算多少、不等读者。

## 3. 落地顺序(给 Kimi)

**前置(若未做)**:Holo 从 Hermes 抽成独立包,`MemoryStore(path)` 可被任意框架打开同一 .db。

1. **① retain 拆事务 + 探重进事务** — 先核现状事务边界。LLM 抽取移出事务、探重移进 `BEGIN IMMEDIATE`。不改:多实例 retain 会超时 + 漏近重复。
2. **② GC 抢锁** — 先把「谁触发 GC」改读为任一实例,再上 `BEGIN IMMEDIATE` 抢锁 + 抢不到就跳。可先用「手动触发 GC」临时解耦,不阻塞灌文档。
3. **③ busy_timeout + checkpoint 降级** — 两个 PRAGMA,最后。

**每步动手前 backup .db**(改造计划 §3-③ 铁律;且 entity/HRR 一致性这类问题靠「动手前快照 vs 动手后 diff」才看得见——已被 HRR 重索引那次验证)。
